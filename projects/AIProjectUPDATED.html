Minimax and Alpha Beta Pruning

We now have a formula that can assign a score to any state of the board, and thus, we can move on to how the computer can use this information to inform what move it should perform. We can do this by using a depth-limited Minimax algorithm. Minimax was chosen because it works well for tic-tac-toe, given that the game is sequential, deterministic, and perfect information.

The process starts with a game state, then creating a tree of all the possible moves from that state, all the moves from those states, and so on. Then, at each step, if the player makes the move, it attempts to minimize the score for the player, and if the computer makes the move, it attempts to maximize the score for the computer. Because our formula above does not require the game to be finished, we can implement our algorithm to work ‘depth limited.’ This means we do not have to simulate the entire tree down to the leaves to run the Minimax algorithm; this immensely increases the algorithm's speed while still keeping it competitive.

The diagram below shows a simple example of the Minimax algorithm for tic-tac-toe.

Alpha-beta pruning was also implemented into the algorithm to improve the performance of the Minimax algorithm. Alpha-beta pruning works by analyzing the previously seen values and selectively removing values from the propagation tree that are guaranteed not to affect the final result.

Here is the sample pseudo code which executes this algorithm:

FUNCTION minimax_alpha_beta(board, limit, alpha, beta, maximizingPlayer)
   gs = CHECK_GAME_STATE(board)
   IF gs IS NOT NULL
       IF gs EQUALS "X"
           RETURN -POWER(board.n, 3), board
       ELSE IF gs EQUALS "O"
           RETURN POWER(board.n, 3), board
       RETURN 0, board

   IF limit EQUALS 0
       RETURN SCORE_OF_BOARD(board), board

   IF maximizingPlayer
       maxEval = NEGATIVE_INFINITY
       maxBoard = NULL
       FOR EACH newBoard IN POSSIBLE_MOVES(board)
           eval, _ = minimax_alpha_beta(newBoard, limit - 1, alpha, beta, FALSE)
           IF eval > maxEval
               maxEval = eval
               maxBoard = newBoard
           alpha = MAX(alpha, eval)
           IF beta <= alpha
               BREAK
       RETURN maxEval, maxBoard
   ELSE
       minEval = INFINITY
       minBoard = NULL
       FOR EACH newBoard IN POSSIBLE_MOVES(board)
           eval, _ = minimax_alpha_beta(newBoard, limit - 1, alpha, beta, TRUE)
           IF eval < minEval
               minEval = eval
               minBoard = newBoard
           beta = MIN(beta, eval)
           IF beta <= alpha
               BREAK
       RETURN minEval, minBoard
END FUNCTION

Automatic Depth Optimization

From executing the algorithm on larger boards, a problem becomes very clear: depth. As mentioned previously, our Minimax function runs depth limited, which greatly increases the performance of the algorithm; however, having a constant depth limit means that it, more often than not, spends too much processing time calculating the first move and not enough calculating the last. The idea behind this section is to create an algorithm that can calculate the maximum depth the algorithm should traverse to, such that it is below a set number of total calculations. This means it will start shallow, only checking a few moves in advance. Since the vast majority of the board is empty at the beginning of the game, a shallow depth will keep performance high, and since not much strategy is needed in the first move, the penalty for this speedup is low. However, as the game progresses, more of the board fills up, reducing the breadth of the tree and allowing the algorithm to check deeper while using the same number of calculations, improving end-game performance. I’d like to discuss why we chose to do a depth-limited, algorithmic approach rather than a full minimax, scoring based on the value of the tree leaves. The primary reason behind this is that we had concerns about the algorithm's speed when it had to simulate the entire tree fully. While speed is mostly a problem within the first turn, we still decided that a depth-limited approach would be more efficient (and fun).
